# Codex 工作流

本文件约束以后使用 Codex 开发时的固定流程。

## 一、固定流程

1. 先读 `AGENTS.md`。
2. 再读 `PROJECT.md`。
3. 再读任务单；没有任务单时，不允许直接大改代码。
4. 根据任务读对应 contract 文件。
5. 搜索相关代码和已有实现。
6. 输出修改计划。
7. 等待高风险操作确认。
8. 执行最小修改。
9. 运行检查命令。
10. 输出修改总结、测试结果、风险点、回滚方式。
11. 提醒人工复核高风险点。

## 二、没有任务单时

- 只能做只读分析、小范围问题定位或创建任务单草稿。
- 不允许直接做大范围重构。
- 不允许修改数据库、支付、权限、订单、状态机等高风险模块。

## 三、需求不清时

- 先列出已知事实。
- 再列出必要假设。
- 对高风险或不可逆操作必须询问用户。
- 禁止为了推进而编造项目事实。

## 四、规范缺失时

- 发现接口规范缺失，先补 `API_CONTRACT.md`。
- 发现数据库规范缺失，先补 `DB_CONTRACT.md`。
- 发现状态机规范缺失，先补 `STATE_MACHINE.md`。
- 发现命名规范缺失，先补 `NAMING_RULES.md`。
- 规范补充必须基于真实项目代码和用户确认，不得凭空设计复杂体系。

## 五、旧代码和规范冲突时

- 先报告冲突文件、冲突规则和影响范围。
- 不要直接强改旧代码。
- 如果任务只要求修 bug，优先用最小改动兼容现状。
- 如果需要迁移到规范，必须先列迁移计划和文件清单。

## 六、修改计划模板

```txt
修改计划：
1. 本次要解决的问题
2. 计划修改的文件
3. 是否影响 API
4. 是否影响数据库
5. 是否影响权限
6. 是否影响状态机
7. 风险点
```

## 七、修改总结模板

```txt
修改总结：
1. 修改了哪些文件
2. 具体改了什么
3. 是否影响 API
4. 是否影响数据库
5. 是否影响权限
6. 是否影响状态机
7. 已运行的检查命令
8. 测试结果
9. 回滚方式
10. 需要人工复核的地方
```
