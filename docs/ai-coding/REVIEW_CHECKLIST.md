# AI 代码审查清单

人工审查 AI 代码时按本清单逐项检查。发现高风险问题时，先暂停合并或发布。

## 一、范围检查

- 是否超出任务范围。
- 是否改了不该改的文件。
- 是否新增无关依赖。
- 是否做了无关重构。
- 是否把简单需求扩展成复杂系统。

## 二、接口检查

- 是否改变接口结构。
- 是否改变统一返回格式。
- 是否改变错误码格式。
- 是否改变分页参数。
- 是否向前台泄露后台字段、Token、密钥、完整卡密、支付配置。

## 三、数据库检查

- 是否改变数据库结构。
- 是否新增、删除、重命名字段。
- 是否新增迁移但没有回滚说明。
- 是否存在 N+1 查询。
- 是否存在无 LIMIT 查询。
- 是否存在无索引高频查询。

## 四、状态机检查

- 是否改变状态机。
- 是否新增或删除状态。
- 是否改变状态含义。
- 是否允许已完成、已退款、已取消等终态发生危险回退。
- 是否存在重复发货、重复扣款、重复加积分风险。

## 五、安全检查

- 是否存在越权风险。
- 是否存在支付漏洞。
- 是否存在敏感字段泄露。
- 是否存在硬编码密钥。
- 是否把密码、Token、密钥、完整卡密写入日志。
- 是否只做前端权限控制而缺少后端校验。

## 六、代码质量检查

- 是否存在吞异常。
- 是否存在未使用代码。
- 是否存在重复函数。
- 是否绕过已有工具函数或封装。
- 是否命名和 `NAMING_RULES.md` 冲突。

## 七、测试检查

- 是否存在测试删库风险。
- 是否通过 lint/typecheck/build/test。
- 是否说明未运行检查的原因。
- 是否覆盖权限、失败、空数据、重复请求等关键路径。
- 是否为了通过测试删除或弱化测试。

## 八、发布和回滚检查

- 是否有清晰回滚方案。
- 是否说明需要人工复核的地方。
- 是否涉及环境变量、数据库迁移、后台配置或部署顺序。
- 是否需要备份数据或配置。
