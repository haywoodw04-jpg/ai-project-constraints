# UI 规则

当前仓库不包含前端页面。本文件是下游项目的 UI 约束模板；如果下游项目有页面或组件，必须根据真实设计和已有代码补全本文件。

## 一、页面布局规范

- 页面结构必须复用项目已有布局。
- 管理后台优先使用稳定的列表、筛选、详情、编辑结构。
- 禁止为了单个页面引入完全不同的布局模式。

## 二、颜色规范

- 颜色必须来自已有设计变量或项目确认的色板。
- 危险、成功、警告、信息状态色必须统一。
- 禁止随意新增相近颜色造成状态含义混乱。

## 三、按钮规范

- 主操作、次操作、危险操作必须有明确视觉区分。
- 危险操作必须二次确认。
- 同类页面的按钮位置和文案保持一致。

## 四、表单规范

- 必填项、默认值、校验错误、提交中状态必须明确。
- 表单提交失败必须展示可理解错误。
- 禁止只做前端校验而缺少服务端校验。

## 五、弹窗规范

- 弹窗只用于确认、轻量编辑或必须阻断的任务。
- 大表单、复杂流程优先使用页面或抽屉，具体形态按项目已有规则确认。
- 禁止同一类操作一会儿弹窗、一会儿侧滑、一会儿新页面。

## 六、表格规范

- 表格必须支持加载态、空状态、错误态。
- 大列表必须分页或虚拟滚动。
- 操作列按钮必须稳定，不因数据加载导致布局跳动。

## 七、分页组件规范

- 分页参数必须与 `API_CONTRACT.md` 保持一致。
- 翻页、筛选、排序变化必须明确是否重置页码。
- 禁止前后端分页参数各用一套命名。

## 八、加载状态规范

- 请求中必须有加载态。
- 重复提交按钮必须禁用或做幂等处理。
- 长任务必须显示进度、状态或可刷新结果。

## 九、空状态规范

- 空状态文案必须说明当前没有什么数据。
- 如果有创建入口，可以提供明确主操作。
- 禁止空白页面没有反馈。

## 十、错误提示规范

- 用户可处理的错误必须提示具体操作建议。
- 权限、登录、网络、校验错误必须区分。
- 禁止把后端堆栈、SQL、密钥路径显示给用户。

## 十一、成功提示规范

- 影响数据的操作成功后必须刷新对应数据或更新本地状态。
- 成功提示不替代真实状态更新。
- 关键操作成功后应可追踪操作记录。

## 十二、移动端适配规范

- 如果项目支持移动端，页面必须在目标宽度下检查布局、文字换行和按钮可点击区域。
- 如果项目不支持移动端，必须在 `PROJECT.md` 标记。

## 十三、组件复用规则

- 新页面优先复用已有组件。
- 新增通用组件前必须确认已有组件不能满足。
- 禁止为单个页面复制一份几乎相同的组件。

## 十四、禁止事项

- 禁止同一类页面一会儿弹窗，一会儿侧滑，一会儿新页面。
- 禁止同一项目混用多套按钮风格。
- 禁止随意引入新的 UI 组件库。
- 禁止随意改变后台管理页布局。
- 禁止新页面不复用已有组件。
- 禁止颜色、间距、圆角、阴影风格随意变化。
