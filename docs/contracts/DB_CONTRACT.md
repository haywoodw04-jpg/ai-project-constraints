# 数据库契约

当前仓库不使用数据库、ORM 或迁移工具。本文件是下游项目的数据库约束模板；如果下游项目新增数据库能力，必须先补全本文件再改 schema。

## 一、ORM 使用规范

- 项目只能使用已确认的 ORM 或数据库访问层。
- 禁止为单个功能绕过现有 ORM 新增一套数据库访问方式。
- 原生 SQL 只能用于 ORM 无法表达或性能明确需要的场景，并必须说明原因。

## 二、数据库迁移规范

- 修改表、字段、索引、约束前必须先输出迁移计划。
- 迁移必须可回滚或说明不可回滚原因。
- 生产迁移前必须先在测试环境验证。
- 禁止未确认直接执行 `ALTER`、`DROP`、`TRUNCATE`、批量 `DELETE`。

## 三、表命名规范

- 表名风格由下游项目统一确认，推荐一种风格后全项目保持一致。
- 禁止同一项目混用复数表名、单数表名和不同大小写风格。

## 四、字段命名规范

- 字段命名必须与 `docs/contracts/NAMING_RULES.md` 一致。
- 时间字段、金额字段、用户 ID、订单号、状态字段必须使用统一命名。
- 禁止新增语义重复字段，例如同时存在 `userId`、`uid` 表示同一含义。

## 五、主键规范

- 每张业务表必须有明确主键。
- 主键类型由项目统一确认。
- 禁止把可变业务字段作为唯一主键，除非业务明确要求且已有唯一约束说明。

## 六、时间字段规范

- 推荐使用 `createdAt`、`updatedAt`、`deletedAt` 或项目已统一的等价字段。
- 时间存储时区必须统一。
- 禁止同一数据库中混用秒、毫秒、字符串时间且无说明。

## 七、金额字段规范

- 金额字段必须声明单位和精度。
- 推荐使用整数最小货币单位保存支付金额，或使用数据库定点数类型。
- 禁止用浮点数保存支付、余额、退款等关键金额。

## 八、软删除规范

- 如果使用软删除，必须统一字段名和查询过滤方式。
- 禁止同一项目混用 `deleted`、`isDeleted`、`deletedAt` 而不说明区别。
- 涉及用户可见数据时，查询必须默认排除软删除记录。

## 九、索引规范

- 高频查询条件必须评估索引。
- 唯一业务键必须用唯一约束保证。
- 禁止高频无索引 LIKE 查询。
- 禁止为低频字段盲目新增索引。

## 十、查询规范

- 列表查询必须分页或限制数量。
- 管理端导出大数据必须走异步任务或分批处理。
- 禁止无 LIMIT 查询大表。
- 禁止 N+1 查询进入核心路径。

## 十一、事务规范

以下逻辑必须使用事务或等效原子机制：

- 订单创建和库存扣减。
- 支付成功和订单状态变更。
- 发货和卡密/库存扣减。
- 余额、积分、会员时长变更。
- 退款和资金状态变更。

## 十二、JSON / JSONB 使用限制

- JSON 字段只保存扩展信息、第三方原始回调、低频配置。
- 核心查询字段不得长期藏在 JSON 内。
- JSON 结构变化必须写明版本或兼容规则。

## 十三、原生 SQL 使用限制

- 原生 SQL 必须参数化，禁止拼接用户输入。
- 必须说明为什么不用 ORM。
- 修改类 SQL 必须说明影响行数和回滚方式。

## 十四、数据删除限制

- 生产数据默认不硬删除。
- 批量删除必须先 dry-run 或备份。
- 测试代码禁止连接生产数据库执行删除。

## 十五、数据库变更审批规则

任何涉及以下内容的变更必须人工确认：

- 新增、删除、重命名字段。
- 修改字段类型、默认值、非空约束。
- 删除表、清空表、覆盖导入。
- 修改订单、支付、库存、余额、积分、卡密相关表结构。

## 十六、禁止事项

- 禁止未确认直接改 schema。
- 禁止未确认直接删除字段或删除表。
- 禁止无 LIMIT 的列表查询。
- 禁止无索引的高频 LIKE 查询。
- 禁止核心业务数据滥用 JSON 字段。
- 禁止混用多套软删除字段。
- 禁止订单、支付、库存、余额、积分、卡密等关键逻辑不使用事务。
